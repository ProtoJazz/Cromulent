## Decisions Made

- Used `binary_id` (UUID7) primary key for `feature_flags` table to be consistent with all other tables in the project
- `get_flags/0` uses `Repo.one(Flags) || %Flags{}` — returns struct defaults without any DB query side effects on fresh install
- `upsert_flags/1` detects insert vs update by checking `id == nil` on the returned struct from `get_flags/0`
- `feature_flags` assign added only in the authenticated branch of `ensure_authenticated` — unauthenticated sessions don't need flags

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] Fixed binary_id primary key for feature_flags table**
- **Found during:** Task 1 (DB migration and schema creation)
- **Issue:** Plan provided migration template using default integer primary key (`create table(:feature_flags) do`). The project uses `binary_id` (UUID7) for all tables, as configured in `config/config.exs` (`migration_primary_key: [type: :binary_id]`). PostgreSQL raised `not_null_violation` on the id column because Ecto's UUID7 generator was not used.
- **Fix:** Updated migration to use `primary_key: false` + explicit `add :id, :binary_id, primary_key: true, null: false`. Updated Flags schema to add `@primary_key {:id, Cromulent.UUID7, autogenerate: true}` and `@foreign_key_type :binary_id`.
- **Files modified:** `priv/repo/migrations/20260302000001_create_feature_flags.exs`, `lib/cromulent/feature_flags/flags.ex`
- **Verification:** `mix ecto.reset && mix ecto.migrate` succeeded; all 9 feature_flags tests pass
- **Committed in:** 47704a4 (Task 1 commit)
