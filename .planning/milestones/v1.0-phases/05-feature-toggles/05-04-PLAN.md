<task type="checkpoint:human-verify" gate="blocking">
  <what-built>
    Complete feature toggle system:
    - DB-backed FeatureFlags context (get_flags/0, upsert_flags/1)
    - feature_flags assign injected into all authenticated LiveViews via ensure_authenticated
    - Voice: sidebar hides voice channels + VoiceChannel.join rejects when disabled
    - Registration: /users/register redirects with flash + Register link hidden on login page
    - Link previews: fetch skipped when disabled (URLs remain clickable links)
    - Email confirmation: confirmation email sent on registration + unconfirmed users blocked from login when required
    - TURN config: DB-backed provider/url/secret replacing System.get_env
    - AdminLive Settings tab: Flowbite toggles + TURN form with Save & Test
    - AdminLive Users tab: Create User form bypasses registration flag
  </what-built>
  <how-to-verify>
    Before testing: `mix phx.server` and ensure you are logged in as an admin.

    **Test 1: Settings tab exists**
    1. Navigate to http://localhost:4000/admin?tab=settings
    2. Expected: Settings tab is active, shows 4 toggle switches (Voice Channels, User Registration, Link Previews, Require Email Confirmation) and a TURN Config section

    **Test 2: Voice disable**
    1. In Settings tab, click the Voice Channels toggle to disable it
    2. Expected: Flash "Setting updated." appears, toggle turns off
    3. Open a new tab or refresh — Expected: voice channels are gone from the sidebar
    4. Re-enable voice, refresh — Expected: voice channels return
    5. (Optional deeper test) Open browser console, try `window.liveSocket.channel.push("voice:channel_id", "phx_join", {})` — Expected: receives error `{reason: "voice_disabled"}`

    **Test 3: Registration disable**
    1. In Settings tab, click the User Registration toggle to disable it
    2. Log out (or use incognito)
    3. Navigate to http://localhost:4000/users/register
    4. Expected: Redirected to /users/log_in with error flash "Registration is closed on this server."
    5. On the login page, Expected: No "Register" link visible
    6. Go back to Admin Settings and re-enable registration

    **Test 4: Admin Create User (bypasses registration flag)**
    1. With registration still disabled (or re-disable it), navigate to http://localhost:4000/admin?tab=users
    2. Fill in the Create User form with a new email/username/password and submit
    3. Expected: New user appears in the users table with flash "User [username] created."
    4. Re-enable registration after testing

    **Test 5: Link previews disable**
    1. In Settings tab, click the Link Previews toggle to disable it
    2. In a text channel, post a URL (e.g. https://example.com)
    3. Expected: URL renders as a plain clickable link — no preview card appears
    4. Re-enable link previews — post same URL — Expected: preview card appears

    **Test 6: TURN config form**
    1. In Settings tab, scroll to TURN Server Configuration
    2. Set Provider to "Coturn (self-hosted)", enter a URL like "turn:localhost:3478", enter a secret
    3. Click "Save & Test"
    4. Expected: Config saved (flash), and a result appears below ("Connection successful" or a failure message — either is acceptable for local test)
    5. Set Provider back to "Disabled", click Save & Test — Expected: no test result shown (disabled)

    **Test 7: Email confirmation (optional — requires email/mailer config)**
    1. In Settings tab, enable "Require Email Confirmation"
    2. Log out, register a new account
    3. Expected: In dev, check http://localhost:4000/dev/mailbox — confirmation email should be in the inbox
    4. Try logging in as the unconfirmed user — Expected: login fails (returns to login page)
    5. If mailbox preview is not available, skip this test and note it

    Report: Confirm all 5 scenarios work, or describe any failures found.
  </how-to-verify>
  <resume-signal>Type "approved" if all scenarios work, or describe specific failures</resume-signal>
